深挖结论

区分「初筛异常」与「根因判断」：初筛显示下游 DB 错误率 50%；深挖确认全部为业务异常。

  • 现象：近 1 小时 service-b 下游 MySQL（demo_apm）错误率 50%，共 109 次错误，116 条失败 Trace。
  • 证据链：入口指标正常（0% 错误率）→ 错误集中在 Dubbo RPC findInventory 调用 → 失败 Trace 全部为 InsufficientStockException → 涉及 SQL SELECT sku,available FROM demo_inventory WHERE sku='DEMO-10001'
  • 可能根因：SKU DEMO-10001 库存不足（置信度高），导致所有查询该 SKU 可用库存的数据库操作均返回空/零库存，触发业务异常。入口层 catch 了该异常并以 200 返回，故入口错误率不受影响。
  • 建议动作：
    1. 🔴 高优先级 — 检查 SKU DEMO-10001 的库存补货情况，确认是否缺货或库存数据未及时同步
    2. 🟡 中优先级 — 评估 InsufficientStockException 的业务处理逻辑，是否需要对用户层给出更明确的提示（当前以 HTTP 200 返回，用户端可能无感知）
    3. 🟢 低优先级 — 为 SKU 设置库存预警阈值，低于阈值时自动触发补货提醒，减少此类异常发生频率
    4. ℹ️ — 基础设施层面（JVM/GC/实例/网络）无需干预，当前服务运行稳定
